# Quicken For Mac Database Schema Reference

> Captured from Quicken 8.5 for Mac, March 2026.
> 
> **No personal financial data is included.** This document contains only structural schema information: table definitions, column names, indexes, entity types, and relationships.

## Core Data Conventions

Quicken for Mac stores its data in a [Core Data](https://developer.apple.com/documentation/coredata) SQLite database located inside the `.quicken` bundle (`data` file). The database follows standard Core Data naming conventions:

- **Z_PK** -- integer primary key on every entity table
- **Z_ENT** -- entity type discriminator (references `Z_PRIMARYKEY.Z_ENT`)
- **Z_OPT** -- optimistic locking counter, incremented on each update
- **Z-prefix** -- all table and column names are prefixed with `Z`
- **TIMESTAMP columns** -- Core Data epoch: seconds since **2001-01-01 00:00:00 UTC**. Add `978307200` to convert to Unix epoch.
- **Z_ENT values are per-database** -- always look up entity type numbers from `Z_PRIMARYKEY` at runtime; they may differ across Quicken data files.
- **Encryption** -- Quicken encrypts the `data` file when the app is closed. The database is only readable while Quicken is running.

## Entity Types (Z_PRIMARYKEY)

The `Z_PRIMARYKEY` table maps entity names to `Z_ENT` integer discriminators. Multiple entities share single physical tables via `Z_ENT` (e.g., `ZTRANSACTION` holds both `CashFlowTransaction` and `InvestmentTransaction`; `ZTAG` holds `CategoryTag`, `UserTag`, and `CashFlowTag`).

## Join Tables (Many-to-Many Relationships)

Core Data uses numbered join tables for many-to-many relationships. The numbers in table and column names correspond to `Z_ENT` values and **vary per database**.

### Z_15USERTAGS

Links `CashFlowTransactionEntry` rows to `UserTag` rows (user-defined tags on transaction splits).

```sql
CREATE TABLE "Z_15USERTAGS" (
    Z_15CASHFLOWTRANSACTIONENTRIES INTEGER,
    Z_76USERTAGS INTEGER,
    PRIMARY KEY (Z_15CASHFLOWTRANSACTIONENTRIES, Z_76USERTAGS)
)
```

Column name mapping (for this particular database snapshot):
- `Z_15CASHFLOWTRANSACTIONENTRIES` -- FK to `ZCASHFLOWTRANSACTIONENTRY.Z_PK`
- `Z_76USERTAGS` -- FK to `ZTAG.Z_PK` where `Z_ENT` = UserTag

### Z_55USERTAGS

Links `QuickFillRuleSplitEntry` rows to `UserTag` rows (tags on auto-fill rules).

```sql
CREATE TABLE "Z_55USERTAGS" (
    Z_55QUICKFILLRULESPLITENTRIES INTEGER,
    Z_76USERTAGS1 INTEGER,
    PRIMARY KEY (Z_55QUICKFILLRULESPLITENTRIES, Z_76USERTAGS1)
)
```

Column name mapping:
- `Z_55QUICKFILLRULESPLITENTRIES` -- FK to `ZQUICKFILLRULESPLITENTRY.Z_PK`
- `Z_76USERTAGS1` -- FK to `ZTAG.Z_PK` where `Z_ENT` = UserTag

### Z_1BUDGETS

Links `Account` rows to `Budget` rows (which accounts are included in which budgets).


## Key Relationships

### Transaction to Account

```sql
SELECT t.*, a.ZNAME AS account_name
FROM ZTRANSACTION t
JOIN ZACCOUNT a ON t.ZACCOUNT = a.Z_PK;
```

- `ZTRANSACTION.ZACCOUNT` references `ZACCOUNT.Z_PK`
- Filter by `ZTRANSACTION.Z_ENT` to select only cash flow or investment transactions

### Transaction to Payee

```sql
SELECT t.*, p.ZNAME AS payee_name
FROM ZTRANSACTION t
JOIN ZUSERPAYEE p ON t.ZUSERPAYEE = p.Z_PK;
```

- `ZTRANSACTION.ZUSERPAYEE` references `ZUSERPAYEE.Z_PK`
- Payee is nullable (investment transactions typically have no payee)

### Transaction Split Entries (Categories)

```sql
SELECT t.Z_PK AS txn_id, e.ZAMOUNT, tag.ZNAME AS category_name
FROM ZTRANSACTION t
JOIN ZCASHFLOWTRANSACTIONENTRY e ON e.ZPARENT = t.Z_PK
JOIN ZTAG tag ON e.ZCATEGORYTAG = tag.Z_PK;
```

- `ZCASHFLOWTRANSACTIONENTRY.ZPARENT` references `ZTRANSACTION.Z_PK`
- `ZCASHFLOWTRANSACTIONENTRY.ZCATEGORYTAG` references `ZTAG.Z_PK` (where `ZTAG.Z_ENT` corresponds to `CategoryTag`)
- A single transaction may have multiple split entries (one per category)
- The sum of split `ZAMOUNT` values equals the transaction total

### Category Tag Hierarchy

```sql
SELECT child.ZNAME AS subcategory, parent.ZNAME AS parent_category
FROM ZTAG child
JOIN ZTAG parent ON child.ZPARENTCATEGORY = parent.Z_PK
WHERE child.Z_ENT = (SELECT Z_ENT FROM Z_PRIMARYKEY WHERE Z_NAME = 'CategoryTag');
```

- `ZTAG.ZPARENTCATEGORY` is a self-referential FK to `ZTAG.Z_PK`
- Top-level categories have `ZPARENTCATEGORY = NULL`
- Hierarchy is typically one level deep (parent > child), though deeper nesting is possible

### Transaction to Transfer Target

```sql
SELECT t.*, target.ZNAME AS transfer_to_account
FROM ZTRANSACTION t
JOIN ZACCOUNT target ON t.ZTARGETACCOUNT = target.Z_PK
WHERE t.ZTARGETACCOUNT IS NOT NULL;
```

- `ZTRANSACTION.ZTARGETACCOUNT` references `ZACCOUNT.Z_PK` for the destination of a transfer
- `ZTRANSACTION.ZSENDACCOUNT` may also be set for the source side

### Investment Positions and Securities

```sql
SELECT p.*, s.ZNAME AS security_name, s.ZTICKER, a.ZNAME AS account_name
FROM ZPOSITION p
JOIN ZSECURITY s ON p.ZSECURITY = s.Z_PK
JOIN ZACCOUNT a ON p.ZACCOUNT = a.Z_PK;
```

- `ZPOSITION.ZSECURITY` references `ZSECURITY.Z_PK`
- `ZPOSITION.ZACCOUNT` references `ZACCOUNT.Z_PK`
- `ZSECURITYID` provides alternate identifiers (CUSIP, ticker) via `ZSECURITYID.ZSECURITY`

### Lot Tracking

```sql
SELECT l.*, lm.ZSHARES, lm.ZCOSTBASIS, t.ZPOSTEDDATE
FROM ZLOT l
JOIN ZLOTMOD lm ON lm.ZLOT = l.Z_PK
JOIN ZTRANSACTION t ON lm.ZTRANSACTION = t.Z_PK;
```

- `ZLOT.ZPOSITION` references `ZPOSITION.Z_PK`
- `ZLOTMOD.ZLOT` references `ZLOT.Z_PK` (modifications to the lot over time)
- `ZLOTMOD.ZTRANSACTION` ties each lot change to a specific transaction
- `ZLOTASSIGNMENT` links source and destination transactions for lot transfers

### Account to Financial Institution

```sql
SELECT a.ZNAME, fi.ZNAME AS institution_name, cfi.ZUSERNAME
FROM ZACCOUNT a
JOIN ZFINANCIALINSTITUTION fi ON a.ZFINANCIALINSTITUTION = fi.Z_PK
LEFT JOIN ZCLOUDFILOGIN cfi ON a.ZCLOUDFILOGIN = cfi.Z_PK;
```

- `ZACCOUNT.ZFINANCIALINSTITUTION` references `ZFINANCIALINSTITUTION.Z_PK`
- `ZACCOUNT.ZCLOUDFILOGIN` references `ZCLOUDFILOGIN.Z_PK` (cloud aggregation credentials)

### Reconciliation

```sql
SELECT rr.ZDATE, rr.ZCLOSINGBALANCE, rt.ZORIGINALTRANSACTION
FROM ZRECONCILERECORD rr
JOIN ZRECONCILEDTRANSACTION rt ON rt.ZRECONCILERECORD = rr.Z_PK;
```

- `ZRECONCILERECORD.ZACCOUNT` references `ZACCOUNT.Z_PK`
- `ZRECONCILEDTRANSACTION.ZRECONCILERECORD` references `ZRECONCILERECORD.Z_PK`
- `ZRECONCILEDTRANSACTION.ZORIGINALTRANSACTION` references `ZTRANSACTION.Z_PK`

## Important Notes

### Shared Physical Tables (Entity Type Inheritance)

Core Data uses **single-table inheritance** for several entity hierarchies. The `Z_ENT` column discriminates subtypes:

| Physical Table | Entity Types | Notes |
|---|---|---|
| `ZTRANSACTION` | Transaction, CashFlowTransaction, SmartCashFlowTransaction, InvestmentTransaction | Investment-specific columns (ZSHARES, ZSHAREPRICE, etc.) are NULL for cash flow rows |
| `ZTAG` | Tag, CategoryTag, UserTag, CashFlowTag | Filter on Z_ENT to get only categories vs. user tags |

### Z79_PARENT Artifact

The `ZTAG` table contains a column `Z79_PARENT` alongside `ZPARENTCATEGORY`. This is a Core Data artifact from model migrations -- the `79` in `Z79_PARENT` references a historical `Z_ENT` value for the `CategoryTag` entity. In practice, use `ZPARENTCATEGORY` for the parent-child category relationship.

### Signed Amounts

- `ZTRANSACTION.ZAMOUNT` -- credits are positive, debits are negative (from the account's perspective)
- `ZCASHFLOWTRANSACTIONENTRY.ZAMOUNT` -- split amounts follow the same sign convention
- Investment transactions use `ZSHARES` (can be negative for sells) and `ZSHAREPRICE` (always positive)

### Date Fields

- `ZTRANSACTION.ZPOSTEDDATE` -- the date the transaction posted; **can be NULL** for pending/scheduled transactions
- `ZTRANSACTION.ZENTEREDDATE` -- when the user entered or the system downloaded the transaction; always populated
- Both are Core Data timestamps (seconds since 2001-01-01)

### Account Type Values

The `ZACCOUNT.ZTYPENAME` column stores account types as **uppercase strings**:

- `CHECKING`, `SAVINGS`, `CREDITCARD`, `CASH`
- `INVESTMENT`, `RETIREMENT` (brokerage/IRA accounts)
- `MORTGAGE`, `LOAN`, `LIABILITY`
- `ASSET`, `PROPERTY`

### ZTAG Multi-Entity Usage

The `ZTAG` table serves triple duty:

1. **CategoryTag** -- income/expense categories (Groceries, Rent, Salary). Has parent-child hierarchy via `ZPARENTCATEGORY`.
2. **UserTag** -- user-created freeform tags. Linked to transactions via the `Z_15USERTAGS` join table.
3. **CashFlowTag** -- system-internal tags for cash flow classification.

Always filter on `Z_ENT` (looked up from `Z_PRIMARYKEY`) when querying `ZTAG` to avoid mixing entity types.

### Quicken Database Access

- The database file is located at `~/Documents/<name>.quicken/data`
- Quicken **encrypts** the file when the application is not running; the `data` file becomes a ~24KB metadata stub
- The file's inode changes when Quicken opens/closes the database
- Always open the database in **read-only mode** (`?mode=ro`) to avoid interfering with Quicken's Core Data stack
